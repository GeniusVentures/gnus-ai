# Phase 08.1: Safe Wallet Proposer retrofit for diamondCut proposals — Context

**Gathered:** 2026-06-29
**Status:** Ready for planning

## Phase Boundary

Retrofit the existing RPC Diamond deployment pipeline so diamondCut/admin transactions are proposed to a Safe wallet (via Safe Transaction Service) instead of executed directly by the deployer EOA. Keep existing facet deployment unchanged — only intercept the final privileged transaction.

This is deployment infrastructure, not a contract change. The diamond itself is not modified.

## Implementation Decisions

### Testing Strategy

- **D-01:** Use Foundry fork tests (`forge test --fork-url <sepolia-rpc>`) against the live Sepolia diamond. Deploy a test Safe (singleton + proxy) in forge setup with a 1-of-2 signer threshold. Transfer diamond ownership to the test Safe address. Forge tests exercise `Safe.execTransaction()` to verify diamondCut executes successfully through the Safe multisig path at the contract level. This proves the full flow (Safe owns diamond → Safe executes diamondCut → facets updated) without external dependencies. Environment defaults: forge/local/Hardhat use EOA or test Safe (no proposal submission); Sepolia/mainnet use the real Safe + TS-level proposal flow.
- **D-02:** Supplement with Hardhat/TS unit tests for the Safe proposal helper (`proposeSafeTransaction.ts`), config validation, and strategy selection logic. Mock the Safe API — do not hit Safe Transaction Service from unit tests. The TS-level Safe SDK integration (proposal submission to Transaction Service) is validated via the Sepolia smoke test, not unit tests.

### Strategy Implementation

- **D-03:** Primary path: subclass `RPCDeploymentStrategy` from `@diamondslab/diamonds` (npm, `github:diamondslab/diamonds`). Override the method that sends contract calls — intercept when target == diamond address AND calldata selector == diamondCut selector.
- **D-04:** If `@diamondslab/diamonds` does not expose a clean override point, fork the library (author is a former team member; fork or write access is available). Do not use provider wrappers or monkey-patching.

### Deployment Record Handling

- **D-05:** Write proposed (not-yet-executed) upgrades to `diamonds/GeniusDiamond/pending-safe-upgrades/<network>-<timestamp>-<safeTxHash>.json`. The canonical `deployments/<network>.json` stays frozen until the Safe tx actually executes on-chain.
- **D-06:** Proposal artifact includes: diamondName, networkName, chainId, safeAddress, proposerAddress, target (diamond address), value ("0"), operation (CALL), safeTxHash, calldata, calldataSelector, origin, createdAt.

### Scope

- **D-07:** Safe proposal mode only applies to existing Diamond upgrades where the Diamond is already deployed and owned/administered by the Safe. Initial deployments are not affected.
- **D-08:** New facet deployments during an upgrade still go through the existing direct RPC flow — only the final diamondCut is intercepted.
- **D-09:** CLI flags: `--safe-propose`, `--safe-address`, `--safe-proposer-private-key` (optional, defaults to PRIVATE_KEY), `--safe-tx-service-url` (optional), `--safe-api-key` (optional), `--safe-origin` (default: `gnus-ai-rpc-upgrade`).
- **D-10:** Env vars: `SAFE_PROPOSE`, `SAFE_ADDRESS`, `SAFE_PROPOSER_PRIVATE_KEY`, `SAFE_TX_SERVICE_URL`, `SAFE_API_KEY`, `SAFE_ORIGIN`.
- **D-11:** Validation when `SAFE_PROPOSE=true`: require SAFE_ADDRESS (valid address), require a proposer private key (64 hex chars with 0x prefix). Do NOT validate that proposer is a Safe owner.
- **D-12:** Mainnet safety: if `networkName === 'mainnet'` and `safePropose !== true`, fail before executing privileged diamondCut. Hardhat/local and Sepolia allowed with loud warning when `SAFE_PROPOSE=false`.

### Claude's Discretion

- Whether to validate that the Safe actually owns the diamond before proposing
- Log level and formatting for proposal confirmation output
- Whether to add a `--dry-run` mode that builds the proposal but doesn't submit

## Existing Code Insights

### Reusable Assets

- `RPCDiamondDeployer` — Already creates strategy, provider, and signer. Strategy selection point is well-defined.
- `scripts/deploy/rpc/common.ts` — Commander-based CLI with `BaseRPCOptions` interface, `createRPCConfig()`, `validateRPCOptions()`. Extension points for new flags already exist.
- `scripts/common.ts` — Contains `diamondCutFuncAbi` — reusable for selector identification.
- `GeniusDiamondTestBase.sol` — Foundry test base. Can be pointed at a fork URL for Sepolia fork tests.

### Established Patterns

- RPC deployment: config → RPCDiamondDeployer → strategy → DiamondDeployer → deployDiamond()
- TypeScript interfaces for config (extendable `RPCDiamondDeployerConfig`)
- Commander `.option()` chain with `process.env` defaults in `common.ts`
- Environment variable validation with accumulated error arrays
- Foundry fork testing already configured in `foundry.toml`

### Integration Points

- `RPCDiamondDeployer.deployDiamond()` → `new DiamondDeployer(this.diamond!, this.strategy).deployDiamond()` — strategy is injected here
- `createRPCConfig(options)` in `common.ts` — maps CLI flags to config object
- `validateRPCOptions(options)` in `common.ts` — validation hook for new Safe flags
- Diamond deployment records at `diamonds/GeniusDiamond/deployments/` — new `pending-safe-upgrades/` sibling directory

## Specific Ideas

- Forge fork tests deploy a 1-of-2 test Safe (singleton + proxy from `@safe-global/safe-contracts`), transfer diamond ownership to Safe, then call `Safe.execTransaction()` with the diamondCut calldata to verify execution. The Safe singleton is at a deterministic address on Sepolia so it's already in the forked state — only the proxy needs deployment.
- Forge/local environments default to EOA or test Safe — no TS-level proposal submission needed. Sepolia and mainnet use the real Safe + TS-level proposal flow.
- `@diamondslab/diamonds` author is a former team member; fork or write access is available if the library needs changes
