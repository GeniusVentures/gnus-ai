## Deviations from Plan

### SDK API Adjustments

**1. [Rule 3 - Blocking] protocol-kit@8.0.1 Safe.init provider type mismatch**
- **Found during:** Task 2
- **Issue:** The plan skeleton assumed `Safe.init({ provider: rpcUrl, ... })` would work, but the installed v8.0.1 type `Eip1193Provider | HttpTransport | SocketTransport` rejects `JsonRpcProvider` instances because ethers v6's provider doesn't match the `RequestArguments` shape. The spec skeleton's original approach of passing the URL string works correctly because `HttpTransport = string` is in the type union.
- **Fix:** Pass `input.rpcUrl` (a URL string) directly as the `provider` value, which satisfies `HttpTransport`. Removed the `JsonRpcProvider` instance pass attempt.
- **Files modified:** scripts/safe/proposeSafeTransaction.ts
- **Commit:** `2da3168`

**2. [Rule 1 - Bug] signHash return type differs from spec skeleton**
- **Found during:** Task 2 (during implementation, before commit)
- **Issue:** Spec skeleton expected `signHash` to return `{ data: string }`, but protocol-kit@8.0.1 returns `SafeSignature` which has `signer`, `data`, and `isContractSignature` fields. Used `senderSignature.data` directly — this matches the spec's expected field name, so no runtime difference.
- **Fix:** No code change needed — the `.data` accessor works identically in both shapes.

**3. [Rule 1 - Bug] OperationType enum reverse-map yields 'Call' not 'CALL'**
- **Found during:** Task 3 (during implementation)
- **Issue:** `OperationType[OperationType.Call]` returns `'Call'` (camelCase from TypeScript enum reverse-map), but the spec artifact shape requires uppercase `'CALL'`.
- **Fix:** Added `.toUpperCase()` when mapping the enum to the artifact's `operation` string field.
- **Files modified:** scripts/safe/writeSafeProposalArtifact.ts
- **Commit:** `f1f3afc`
