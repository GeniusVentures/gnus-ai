## Forge Testing Strategy

| Option           | Description                                                                                             | Selected |
| ---------------- | ------------------------------------------------------------------------------------------------------- | -------- |
| Smoke test only  | Manual Sepolia smoke test with SAFE_PROPOSE=true, no automated fork tests                               |          |
| Forge fork tests | `forge test --fork-url <sepolia-rpc>` — forks live Sepolia chain state for testing against real diamond | ✓        |
| Hardhat only     | Existing Hardhat/TS unit test infrastructure, no forge                                                  |          |

**User's choice:** Forge fork tests against the live Sepolia diamond. Smoke test alone is insufficient — it can't reproduce the exact deployed state (facet versions, storage, selector mismatches) that the real diamond has.

**Notes:** Forge fork testing catches facet version mismatches, storage layout conflicts, and selector collisions that only surface against the actual deployed state. Supplemented with Hardhat/TS unit tests for the TypeScript Safe helper code.

## Strategy Implementation

| Option                         | Description                                                           | Selected |
| ------------------------------ | --------------------------------------------------------------------- | -------- |
| Subclass RPCDeploymentStrategy | Extend the class from @diamondslab/diamonds, override the send method | ✓        |
| Provider wrapper               | Wrap ethers.JsonRpcProvider to intercept broadcastTransaction()       |          |
| Monkey-patch                   | Replace strategy.sendTransaction on the instance after creation       |          |

**User's choice:** Subclass with fallback. If @diamondslab/diamonds doesn't expose a clean override point, fork the library rather than using wrapper or monkey-patch approaches.

**Notes:** @diamondslab/diamonds is `github:diamondslab/diamonds` on npm. The author is a former team member — fork or write access is available. Subclassing is the cleanest path; forking is the fallback. Provider-level wrapping and monkey-patching are rejected as too fragile.

## Deployment Record Handling

| Option                          | Description                                                                            | Selected |
| ------------------------------- | -------------------------------------------------------------------------------------- | -------- |
| Pending-safe-upgrades directory | `diamonds/GeniusDiamond/pending-safe-upgrades/<network>-<timestamp>-<safeTxHash>.json` | ✓        |
| Proposal artifact only          | Write only the Safe proposal JSON, no directory structure                              |          |

**User's choice:** Pending-safe-upgrades directory. Keeps canonical deployment records frozen until Safe tx executes on-chain. Provides a discoverable location for outstanding proposals.

**Notes:** Proposal artifact includes diamondName, networkName, chainId, safeAddress, proposerAddress, target, calldata, safeTxHash, timestamp. Manual reconciliation after Safe execution is acceptable.

## Forge Safe Environment Routing

| Option                    | Description                                                                                                                                    | Selected |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- | -------- |
| EOA default for Forge     | forge/local/Hardhat use deployer EOA, Safe SDK tested via TS mocks only                                                                        |          |
| Deploy test Safe in Forge | forge fork tests deploy a 1-of-2 test Safe, transfer diamond ownership, exercise Safe.execTransaction() to verify diamondCut at contract level | ✓        |

**User's choice:** Deploy a test Safe (singleton + proxy) in forge fork setup with 1-of-2 signer threshold. Forge tests exercise `Safe.execTransaction()` to verify diamondCut executes through the multisig path. The Safe singleton is at a deterministic address on Sepolia — already in the forked state. Only the proxy needs deployment.

**Environment routing decided:**

- Forge/local/Hardhat → EOA or test Safe (no TS-level proposal submission)
- Sepolia/mainnet → real Safe + TS-level proposal flow

**Notes:** This proves the contract-level multisig path works before any TS API integration. TS-level Safe SDK tests are separate (Hardhat mocks + Sepolia smoke test).
