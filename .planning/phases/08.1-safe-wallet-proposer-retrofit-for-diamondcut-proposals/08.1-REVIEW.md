# Phase 08.1: Code Review Report

**Reviewed:** 2026-06-29T00:00:00Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

The Safe-proposer retrofit for `diamondCut` proposals is a focused, well-scoped change.
The type contracts are clean, the strategy override correctly bypasses the canonical
deployment-data update (the upgrade is "proposed," not "executed"), and the unit tests
cover the strategy selection matrix and Safe SDK call shape.

However, several **blocker** issues need fixing before this code ships:

1. **The Safe proposer will not actually sign Safe proposals correctly** —
   `signHash` returns an ECDSA signature over the Safe tx hash, but a Safe proposal
   submitted via the Transaction Service must also satisfy the Safe's `safeNonce`
   semantics; the implementation passes `safeTransaction.data` (the unsigned
   `SafeTransactionData`) to `proposeTransaction` without first invoking
   `protocolKit.signTransaction()`. This is the single biggest risk — the proposal
   may be silently rejected or appear as an unsigned transaction in the Safe UI.
2. **`proposeSafeTransaction` does not validate the `to`/`safeAddress` inputs**
   before submitting — a malformed or zero address can produce a proposal that
   signers later execute against the wrong contract.
3. **The mainnet guard is bypassable via `networkName` capitalization / whitespace**
   (e.g., `"Mainnet"`, `" mainnet"`), because the comparison uses strict `===`
   against the literal string `"mainnet"`.
4. **`SafeProposerRPCDeploymentStrategy.performDiamondCutTasks` will propose a
   no-op `diamondCut` when `facetCuts` is empty** — the OZ strategy and the parent
   RPC strategy both short-circuit on empty cuts; this override does not, so it
   will create a spurious Safe proposal.

Additional warnings cover redundant calls to `this.getConfig().rpcUrl` (the strategy
already stores everything it needs), private-key logging risk in verbose mode, the
`initAddress === ZeroAddress` warning from `getInitCalldata` that the override
suppresses by never logging it, and a few test-quality gaps (no assertion that the
`safeNonce`/`safeTransactionData` shape is correct, no coverage for the empty-cuts
case, stubs that bypass `proposeTransaction` argument assertions).

### Critical Issues

### CR-01: Safe proposal is submitted without a properly signed Safe transaction

**File:** `scripts/safe/proposeSafeTransaction.ts:69-91`
**Issue:**

The flow builds a Safe transaction via `protocolKit.createTransaction(...)`,
computes its hash via `getTransactionHash(...)`, then calls `protocolKit.signHash(safeTxHash)`
and submits `safeTransaction.data` (the **unsigned** `SafeTransactionData`) plus the
standalone signature bytes to `apiKit.proposeTransaction`.

`signHash` in `@safe-global/protocol-kit` produces an off-chain signature, which is
the correct primitive for adding an *additional* signature to a proposal, BUT it
**requires the transaction to already have the proposer's signature appended to the
SafeTransaction object** when the Safe service reconstructs and validates the
proposal. The canonical protocol-kit pattern is:

```ts
const safeTransaction = await protocolKit.createTransaction({ transactions: [...] });
const signedSafeTransaction = await protocolKit.signTransaction(safeTransaction);
const safeTxHash = await protocolKit.getTransactionHash(signedSafeTransaction);
await apiKit.proposeTransaction({
    safeAddress,
    safeTransactionData: signedSafeTransaction.data, // .data still carries signatures via .signatures
    safeTxHash,
    senderAddress,
    senderSignature: senderSignature.data,
    origin,
});
```

By skipping `signTransaction` and passing the unsigned `safeTransaction.data`, the
proposal will appear in the Safe UI as a transaction where the proposer's signature
is not part of the encoded payload. Depending on the Safe service version, this can
result in:

- The signature being silently discarded (proposal appears as "needs confirmation"
  even from the proposer)
- An HTTP 422 from the Safe Transaction Service (`SafeSignature not valid` /
  `Signatures not sorted`), which would surface as an unhandled promise rejection
  from `apiKit.proposeTransaction` and abort the upgrade flow with no artifact written.

The unit tests stub `SafeApiKit.proposeTransaction` to `resolves()`, so this defect
is **invisible to the test suite**.

**Fix:**

```ts
const safeTransaction = await protocolKit.createTransaction({
    transactions: [safeTransactionData],
});

// Sign the Safe transaction properly so the signature is embedded.
const signedSafeTransaction = await protocolKit.signTransaction(safeTransaction);

const safeTxHash = await protocolKit.getTransactionHash(signedSafeTransaction);

const senderSignature = await protocolKit.signHash(safeTxHash);

await apiKit.proposeTransaction({
    safeAddress: input.safeAddress,
    safeTransactionData: signedSafeTransaction.data,
    safeTxHash,
    senderAddress: proposerAddress,
    senderSignature: senderSignature.data,
    origin: input.origin || 'gnus-ai-rpc-upgrade',
});
```

If `signHash` alone is confirmed correct for the installed protocol-kit version
(verify against the SDK's own `propose` example), at minimum add an integration
assertion that `safeTransactionData` carries the proposer's signature, or that
`proposeTransaction` is called with the signed variant. Otherwise treat this as a
blocker and verify end-to-end on Sepolia before mainnet.

### CR-02: No input validation on `to`, `safeAddress`, or `value` in `proposeSafeTransaction`

**File:** `scripts/safe/proposeSafeTransaction.ts:35-67`
**Issue:**

`proposeSafeTransaction` accepts arbitrary strings for `to`, `safeAddress`, `value`,
and `data` and passes them straight to the Safe SDK and ethers `JsonRpcProvider`.
`RPCDiamondDeployerConfig` validation only checks `safeAddress` via `ethers.isAddress`,
but `to` (the Diamond target) and the calldata never reach `validateConfig` — they
are produced by the strategy at runtime.

A bad `to` address (typo, leading space, mixed-case without checksum, `undefined`
from a failed `diamond.getDeployedDiamondData().DiamondAddress` lookup) would result
in a Safe proposal whose `to` field the Safe signers will execute against. Because
the canonical deployment data file is intentionally NOT updated, the local artifact
becomes the only source of truth — and a wrong `to` baked into that artifact is a
silent failure that may not be caught until execution.

`value` is also accepted unvalidated — passing `"abc"` or `"1.5ether"` would propagate
to the Safe transaction and the artifact.

**Fix:**

```ts
export async function proposeSafeTransaction(
    input: ProposeSafeTransactionInput,
): Promise<ProposeSafeTransactionResult> {
    // Validate target and Safe addresses
    if (!ethers.isAddress(input.to)) {
        throw new Error(`proposeSafeTransaction: invalid 'to' address: ${input.to}`);
    }
    if (!ethers.isAddress(input.safeAddress)) {
        throw new Error(`proposeSafeTransaction: invalid 'safeAddress': ${input.safeAddress}`);
    }
    // Validate calldata shape (0x-prefixed hex)
    if (!/^0x[0-9a-fA-F]*$/.test(input.data)) {
        throw new Error("proposeSafeTransaction: 'data' must be 0x-prefixed hex");
    }
    // Validate value parses as wei
    if (input.value !== undefined) {
        try {
            BigInt(input.value);
        } catch {
            throw new Error(`proposeSafeTransaction: 'value' must be a wei integer: ${input.value}`);
        }
    }
    // ...rest of function
}
```

Mirror these checks in `writeSafeProposalArtifact` before persisting the artifact
(the artifact becomes the source of truth, so it must not store unvalidated input).

### CR-03: Mainnet guard is bypassable via `networkName` casing / whitespace

**File:** `scripts/setup/RPCDiamondDeployer.ts:550` (and `scripts/setup/strategies/SafeProposerRPCDeploymentStrategy.ts:131`)
**Issue:**

Both mainnet guards compare `networkName === 'mainnet'` with strict equality. The
`networkName` field originates from:

- CLI flag `--network-name` (free text)
- `process.env.NETWORK_NAME` (free text)
- Hardhat `chainManager.chains[networkName]` lookup key (free text)

A user or CI pipeline passing `--network-name Mainnet`, `--network-name MAINNET`,
`--network-name " mainnet"`, or `--network-name ethereum-mainnet` would **bypass the
hard mainnet guard** that is the primary safety mechanism preventing direct
`diamondCut` execution on mainnet. The Sepolia advisory on line 606 has the same
weakness.

This is the central safety invariant of the whole change. The guard must not be
defeatable by case or whitespace.

**Fix:**

```ts
// In RPCDiamondDeployer.validateConfig:
const normalizedNetwork = (config.networkName || '').trim().toLowerCase();
if (normalizedNetwork === 'mainnet' && !config.safePropose) {
    errors.push(
        'Mainnet privileged Diamond cut/admin upgrades require SAFE_PROPOSE=true — refusing to deploy directly',
    );
}

// In SafeProposerRPCDeploymentStrategy.performDiamondCutTasks:
if (this.networkName.trim().toLowerCase() === 'mainnet') {
    console.warn(chalk.yellow('🛡️  DIAMOND CUT VIA SAFE: ...'));
}

// In deployDiamond (Sepolia advisory):
if (this.networkName.trim().toLowerCase() === 'sepolia' && !this.config.safePropose) {
    ...
}
```

Consider normalizing `networkName` once in the constructor / config builder so every
downstream comparison uses the canonical form.

### CR-04: Safe proposer will create a no-op proposal when there are zero facet cuts

**File:** `scripts/setup/strategies/SafeProposerRPCDeploymentStrategy.ts:166-199`
**Issue:**

`performDiamondCutTasks` unconditionally proceeds to build calldata, call
`proposeSafeTransaction`, and write an artifact even when `facetCuts.length === 0`.

Compare to the OZ Defender strategy (in the same package), which short-circuits:

```js
// node_modules/@diamondslab/diamonds/dist/strategies/OZDefenderDeploymentStrategy.js:527-532
if (facetCuts.length === 0) {
    if (this.verbose) {
        console.log(chalk.yellow('⏩ No DiamondCut operations needed - all facets already deployed and up to date'));
    }
    return;
}
```

For an idempotent re-run, a CI pipeline re-proposing after signers rejected an
earlier proposal, or a no-op upgrade, this override will create a Safe proposal
containing `diamondCut([], ZeroAddress, '0x')` — a no-op transaction that Safe
signers are then asked to review and execute. This wastes signer attention and
produces a misleading artifact that suggests an upgrade happened.

**Fix:**

```ts
const facetCuts: FacetCuts = await this.getFacetCuts(diamond);
await this.validateNoOrphanedSelectors(facetCuts);

if (facetCuts.length === 0) {
    console.log(
        chalk.yellow('⏩ No DiamondCut operations needed — no Safe proposal will be created.'),
    );
    return;
}
```

### WR-01: `SafeProposerRPCDeploymentStrategy` makes a redundant `this.getConfig()` call that returns a Promise for `signerAddress`

**File:** `scripts/setup/strategies/SafeProposerRPCDeploymentStrategy.ts:189`
**Issue:**

`this.getConfig().rpcUrl` is used to pass the RPC URL into `proposeSafeTransaction`.
But `getConfig()` on the parent `RPCDeploymentStrategy` returns an object whose
`signerAddress` field is `Promise<string>` (see the type declaration in
`node_modules/@diamondslab/diamonds/dist/strategies/RPCDeploymentStrategy.d.ts:127-134`).
`getConfig()` is not free — it constructs an object and a Promise every call.

More importantly, the strategy already has all the information it needs in its own
constructor-stored fields (`safeAddress`, `chainId`, etc.) but does NOT store the
`rpcUrl` — it relies on the parent's `private rpcUrl` and re-extracts it via
`getConfig()`. This is fragile: if the parent ever stopped exposing `rpcUrl` through
`getConfig()`, the override breaks silently.

**Fix:** Store `rpcUrl` in the subclass constructor (like the other config fields)
and use `this.rpcUrl` directly:

```ts
private readonly rpcUrl: string;
// ...
constructor(config: SafeProposerStrategyConfig) {
    super(...);
    this.rpcUrl = config.rpcUrl;
    // ...
}

// In performDiamondCutTasks:
rpcUrl: this.rpcUrl,
```

### WR-02: `verbose` mode will log deployer/proposer private-key-derived addresses unconditionally; no redaction of `rpcUrl` query-string API keys

**File:** `scripts/setup/RPCDiamondDeployer.ts:219-225`, `scripts/setup/RPCDiamondDeployer.ts:615-623`, `scripts/setup/strategies/SafeProposerRPCDeploymentStrategy.ts:217-242`
**Issue:**

`verbose: true` prints `🔗 RPC URL: ${config.rpcUrl}` (line 622). RPC URLs in this
project routinely embed Infura/Alchemy project IDs as query-string parameters
(`?api_key=...`). Verbose logging will leak these into CI logs.

Separately, the Safe-proposal success block (strategy lines 217-242) prints
`👤 Proposer: ${result.proposerAddress}` — this is fine, but the surrounding code
path also has access to `this.proposerPrivateKey` and there is no test or lint guard
ensuring the private key never makes it into a log line. The current code does not
log the key, but the surface area is unguarded.

**Fix:** Strip query-string parameters from the RPC URL before logging:

```ts
function redactRpcUrl(url: string): string {
    try {
        const u = new URL(url);
        u.search = ''; // drop ?api_key=...
        return u.toString();
    } catch {
        return url; // not a URL, log as-is
    }
}
// then: console.log(chalk.blue(`🔗 RPC URL: ${redactRpcUrl(config.rpcUrl)}`));
```

### WR-03: Empty `catch {}` swallows real errors from `getContractArtifact`

**File:** `scripts/setup/strategies/SafeProposerRPCDeploymentStrategy.ts:145-157`
**Issue:**

```ts
try {
    const diamondCutArtifact = await getContractArtifact('IDiamondCut', diamond);
    diamondCutAbi = diamondCutArtifact.abi;
} catch {
    // Fall through — use the local ABI fragment
    ...
}
```

The catch is empty (no error binding). It silently swallows all errors — including
non-recoverable ones like a corrupted artifact file or a permissions error — and
falls through to the local ABI. If the local ABI happens to be compatible (which it
currently is), the upgrade proceeds without any indication that the canonical
artifact resolution failed. This is the exact "empty catch" anti-pattern flagged
in the quick-scan checklist.

**Fix:**

```ts
try {
    const diamondCutArtifact = await getContractArtifact('IDiamondCut', diamond);
    diamondCutAbi = diamondCutArtifact.abi;
} catch (error) {
    if (this.verbose) {
        console.log(
            chalk.gray(
                `ℹ️  IDiamondCut artifact not resolved via Hardhat (${(error as Error).message}); using local ABI fragment.`,
            ),
        );
    }
}
```

Binding the error lets verbose mode show *why* resolution failed, distinguishing
"Hardhat not available" (expected) from "artifact file corrupted" (actionable).

### WR-05: `createConfigFromEnv` env-var lookup is buggy for multi-word overrides

**File:** `scripts/setup/RPCDiamondDeployer.ts:459-467`
**Issue:**

```ts
const requiredEnvVars = ['RPC_URL', 'PRIVATE_KEY', 'DIAMOND_NAME'];
for (const envVar of requiredEnvVars) {
    if (
        !process.env[envVar] &&
        !overrides?.[
            envVar.toLowerCase().replace('_', '') as keyof RPCDiamondDeployerConfig
        ]
    ) {
        throw new Error(`Missing required environment variable: ${envVar}`);
    }
}
```

`envVar.toLowerCase().replace('_', '')` only removes the **first** underscore. For
`RPC_URL` this yields `'rpcurl'` (not `rpcUrl`), so the override lookup never matches
the actual `rpcUrl` key on `RPCDiamondDeployerConfig`. For `PRIVATE_KEY` it yields
`'privatekey'` (not `privateKey`). The check therefore only ever validates the env
var; the override branch is dead. A caller relying on `createConfigFromEnv({ rpcUrl:
... })` without setting `process.env.RPC_URL` will get a spurious "Missing required
environment variable: RPC_URL".

Use a regex (`/\_/g`) or an explicit map. This is pre-existing code, but it directly
affects the Safe-proposal CLI flow where a user might pass overrides for `safeAddress`
or `rpcUrl` and expect validation to acknowledge them.

**Fix:**

```ts
const ENV_TO_CONFIG_KEY: Record<string, keyof RPCDiamondDeployerConfig> = {
    RPC_URL: 'rpcUrl',
    PRIVATE_KEY: 'privateKey',
    DIAMOND_NAME: 'diamondName',
};

for (const envVar of requiredEnvVars) {
    const configKey = ENV_TO_CONFIG_KEY[envVar];
    if (!process.env[envVar] && !overrides?.[configKey]) {
        throw new Error(`Missing required environment variable: ${envVar}`);
    }
}
```

### WR-06: `proposeSafeTransaction` leaks the proposer's private key through the `ethers.Wallet` even when only the address is needed

**File:** `scripts/safe/proposeSafeTransaction.ts:39-41`
**Issue:**

```ts
const provider = new ethers.JsonRpcProvider(input.rpcUrl);
const proposer = new ethers.Wallet(input.proposerPrivateKey, provider);
const proposerAddress = await proposer.getAddress();
```

The `proposer` wallet is constructed and bound to a live provider (which keeps a
network connection open), but only `proposerAddress` is ever read from it. The
actual signing is delegated to `protocolKit` via `Safe.init({ signer:
input.proposerPrivateKey })` on line 56-59. The `proposer` wallet is therefore
dead weight: it opens a second provider connection (the protocol kit opens its own)
and holds the private key in memory longer than necessary.

This is a robustness/hygiene issue rather than a security hole, but it does mean
the function makes two RPC connections per call instead of one, doubling the
connection-failure surface.

**Fix:**

```ts
// Derive the proposer address without instantiating a second provider+wallet.
const proposerAddress = ethers.computeAddress(input.proposerPrivateKey);
```

`computeAddress` derives the address from the key with no network I/O. The provider
binding can be removed entirely.

### WR-07: `validateConfig` regex for private keys is too strict — rejects valid 0x-less and mixed-case keys

**File:** `scripts/setup/RPCDiamondDeployer.ts:515, 544`
**Issue:**

Both the main private-key check and the Safe proposer-key check use
`/^0x[a-fA-F0-9]{64}$/`. ethers `Wallet` and `computeAddress` accept private keys
without the `0x` prefix and with any mix of hex case. The validation regex
**requires** lowercase-or-uppercase 64-hex with mandatory `0x`. A user pasting a
key copied from a wallet export that omits the prefix will hit a confusing
"Configuration validation failed" error.

This is also the only validation surface — a too-strict regex means a working key
is rejected, and the test suite only exercises the canonical hardhat-#0 key, so the
false-rejection path is never covered.

**Fix:**

```ts
const PRIVATE_KEY_RE = /^(0x)?[a-fA-F0-9]{64}$/;
// ...
if (!config.privateKey.match(PRIVATE_KEY_RE)) {
    errors.push('Private key must be 64 hex characters (0x prefix optional)');
}
```

Note: `ethers.isAddress` already handles address normalization; mirror that leniency
for keys.

### IN-01: `OperationType[params.operation].toUpperCase()` relies on TypeScript reverse-mapping semantics

**File:** `scripts/safe/writeSafeProposalArtifact.ts:63`
**Issue:**

`OperationType` is a numeric enum from `@safe-global/types-kit`. Numeric enums get
a reverse map (`OperationType[0] === 'Call'`). The code uppercases the result to
produce `"CALL"`. This works today, but if `@safe-global/types-kit` ever switches
`OperationType` to a `const enum` or a string enum, the reverse map disappears and
`OperationType[params.operation]` returns `undefined` — `.toUpperCase()` then throws
at runtime while writing the audit-trail artifact.

**Fix:** Use an explicit map to remove the dependency on enum-implementation
details:

```ts
const OPERATION_STR: Record<OperationType, string> = {
    [OperationType.Call]: 'CALL',
    [OperationType.DelegateCall]: 'DELEGATECALL',
};
const operationStr = OPERATION_STR[params.operation];
```

### IN-02: `calldataSelector` slice silently returns garbage on short calldata

**File:** `scripts/safe/writeSafeProposalArtifact.ts:58`
**Issue:**

`params.calldata.slice(0, 10)` assumes calldata is at least 10 chars (`0x` + 4
bytes). If calldata is shorter (e.g. `"0x"` for a no-arg call that was somehow
empty, or a malformed input that slipped past validation), the slice returns the
whole string and the artifact stores a misleading `calldataSelector`. CR-02 adds
upstream validation, but the writer should also defend itself.

**Fix:**

```ts
const calldataSelector = params.calldata.length >= 10
    ? params.calldata.slice(0, 10)
    : params.calldata;
```

### IN-07: Strategy test stub for `getConfig` returns `signerAddress: Promise.resolve(...)`, papering over a real bug

**File:** `test/unit/rpc/SafeProposerRPCDeploymentStrategy.test.ts:99-106`
**Issue:**

The stub for `getConfig` returns an object with `signerAddress: Promise.resolve(VALID_SAFE_ADDRESS)`.
In production, `getConfig()` returns `{ signerAddress: Promise<string>, ... }` because
the parent calls `this.signer.getAddress()` lazily. This means any code path that
*awaits* `getConfig().signerAddress` works in tests but in production depends on the
parent signer being initialized — which, for `SafeProposerRPCDeploymentStrategy`,
uses the deployer's `privateKey`, not the proposer's. The override never reads
`signerAddress` from `getConfig`, so this is not a live bug today, but the test
stub masks the real return-type shape (a Promise, not a string). If a future change
touches this path the test will not catch the regression.

**Fix:** Replace the inline stub with the real `getConfig()` shape or add a
type-level assertion that the stub matches the parent's return type.
