`verifyFacets.ts`), the Etherscan V2 single-key config, and a delegate-proposer
ECDSA signing branch in `proposeSafeTransaction.ts`. The ECDSA signing flow and

## CR-01: `verifyFacets.ts` path resolver checks `existsSync` on a `path:Name` string — always false, so no facet is ever verified

**Location:** `scripts/verify/verifyFacets.ts:29-39`

```ts
function resolveContractPath(facetName: string): string | null {
    const primary = `contracts/gnus-ai/${facetName}.sol:${facetName}`;
    if (existsSync(primary)) return primary;          // ← BUG
    const starter = `contracts-starter/contracts/facets/${facetName}.sol:${facetName}`;
    if (existsSync(starter)) return starter;          // ← BUG
    return null;
}
```

**Problem.** The `path:ContractName` form is forge/Foundry source-unit syntax,
not a filesystem path. `existsSync('contracts/gnus-ai/GeniusOwnershipFacet.sol:GeniusOwnershipFacet')`
looks for a file literally named `GeniusOwnershipFacet.sol:GeniusOwnershipFacet`,
which does not exist. **Verified empirically:** the colon form returns `false`
while the plain `.sol` returns `true`. The `contracts-starter/` fallback is also
dead — that directory does not exist in the repo at all.

**Impact.** Every facet hits the `⚠️ no contract file found — skipping` branch;
`updated` stays 0; the deployed-data file is never written; the script reports
`0 facets verified`. The script cannot verify anything.

**Fix.** Check the filesystem for the `.sol` file, then append the `:Name` forge
reference only for the return value:

```ts
function resolveContractPath(facetName: string): string | null {
    const primaryFile = `contracts/gnus-ai/${facetName}.sol`;
    if (existsSync(primaryFile)) return `${primaryFile}:${facetName}`;
    const starterFile = `contracts-starter/contracts/facets/${facetName}.sol`;
    if (existsSync(starterFile)) return `${starterFile}:${facetName}`;
    return null;
}
```

## CR-02: `confirmDeployment.ts` hand-rolled ABI decoder has a byte-offset vs word-index bug — produces garbage on real calldata

**Location:** `scripts/safe/confirmDeployment.ts:57-93` (`decodeFacetCuts`)

**Problem.** `readWord(offset)` slices `data.slice(offset*2, (offset+32)*2)`, so
its argument is a **byte** offset. But callers pass **word** indices:

- `arrayOffset = parseInt(readWord(0),16) / 32` → `3` (a word index), then
  `readWord(arrayOffset)` reads byte-offset `3` instead of byte-offset `96`.
- `cursor = arrayOffset + 1` then `readWord(cursor)` repeats the error.
- `selOffset = … / 32` (word index) is later passed to `readWord` the same way.

**Verified empirically** against the real proposal artifact
(`sepolia-11155111-1782951306449-0x2cc4….json`): the buggy decoder returns
`arrayLen = 1610612736` (it misreads `0x60` at the wrong position), whereas
`ethers.Interface.decodeFunctionData` decodes the same calldata cleanly to
**22 FacetCuts**.

**Impact.** With real calldata the function returns nonsense (or the
`for (let i = 0; i < 1610612736; i++)` loop runs until OOM/timeout). The script
cannot run end-to-end; `sepolia.json` was updated by hand instead.

**Fix.** The repo already maintains a correct IDiamondCut ABI in
`scripts/setup/strategies/diamondCutEncoding.ts:30-55`. Decode with
`ethers.Interface` instead of offset math, and drop `decodeFacetCuts` entirely:

```ts
import { ethers } from 'ethers';
// reuse the same kIDiamondCutAbi shape as diamondCutEncoding.ts
const kIDiamondCutAbi = [ /* …diamondCut(tuple[],address,bytes)… */ ];

const decoded = new ethers.Interface(kIDiamondCutAbi).decodeFunctionData(
    'diamondCut', proposal.calldata,
);
const cuts: FacetCut[] = decoded[0].map((c: any) => ({
    facetAddress: c.facetAddress,
    action: Number(c.action),
    functionSelectors: c.functionSelectors.map((s: string) => ethers.hexlify(s)),
}));
```

## WR-01: `verifyFacets.ts` marks `verified: true` on submit, not on actual verification

**Location:** `scripts/verify/verifyFacets.ts:98-103`

**Problem.** `forge verify-contract` (without `--watch`) returns exit `0` as soon
as the verification request is *submitted* — it does not wait for Etherscan to
compile and confirm. The script sets `facets[name].verified = true` immediately,
so `verified` means "submitted", not "verified". A later Etherscan failure
(compiler mismatch, constructor args, etc.) would leave the facet falsely marked
verified.

**Fix.** Either run with `--watch` (blocks until Etherscan resolves, throws on
failure) or follow with `forge verify-check <guid> --chain-id …` and only set
`verified = true` on a successful check.

## WR-02: `confirmDeployment.ts` records `safeTxHash` as the on-chain `tx_hash`

**Location:** `scripts/safe/confirmDeployment.ts:150, 204`

**Status:** ✅ Resolved (redesigned). Instead of patching the L1 `tx_hash` into
`confirmDeployment.ts`, WR-02 is delivered by a new coarse confirmation script,
`scripts/safe/checkSafeExecuted.ts` (npm: `yarn check:safe-executed`).

**Problem.** The comment at `:148-150` admits it: "we don't have it from the
proposal artifact — use the safeTxHash for tracking". But `safeTxHash` is the
Safe-internal transaction hash (hash of the Safe tx data), while every other
entry's `tx_hash` holds the L1 execution transaction hash. The two are
unrelated; consumers that index by `tx_hash` will point at a non-existent L1
transaction.

**Resolution.** `checkSafeExecuted.ts` polls
`apiKit.getTransaction(safeTxHash)` and, on execution, records the real on-chain
`transactionHash` as a top-level `lastExecTxHash` on the deployed-data file
(the deployed-data file is the upgrade source of truth; it does NOT read the
loupe or rebuild `FacetDeployedInfo`). When the Safe has not executed within the
wait window, it stamps a top-level `pending: true` + `pendingSafeTxHash` and
exits 2; `--time-out 0` performs a single check with no wait. Verified
end-to-end against the live sepolia proposal
`0x2cc4…`, which is executed with L1 tx
`0xe487621028092212f5ecd86646962981f8d19b9e071acc18640577594a763d8c` (nonce 4).

## WR-03: `confirmDeployment.ts` decoder output is dead code; `decodeFacetCuts` is unused risk

**Location:** `scripts/safe/confirmDeployment.ts:165-175, 185`

**Problem.** `cutAddrToAction` is built from the decoded cuts, then in the
new-facet branch `const cutInfo = cutAddrToAction[addrLower]` is assigned and
**never used** — actual naming falls through to the on-chain selector-overlap
match at `:187-194`. The on-chain loupe (`diamond.facets()`) already returns the
authoritative post-upgrade facet→selector map, so the manual decode contributes
nothing but risk (see CR-02).

**Fix.** After CR-02, decide whether the decoded action info is needed at all.
If not, remove `decodeFacetCuts`, `cutAddrToAction`, and the dead `cutInfo`
line. If actions are needed (e.g. to log Added vs Replaced), decode via ethers
(CR-02 fix) and actually consume them.

## WR-04: `confirmDeployment.ts` has no chain-id guard — proposal and RPC can be for different chains

**Location:** `scripts/safe/confirmDeployment.ts:122-133`

**Problem.** `proposal.chainId` (from the artifact) is never compared to the
chain the `--rpc-url` provider is actually connected to. Pointing at mainnet RPC
with a sepolia proposal artifact would check the wrong Safe's nonce and silently
report "not executed" (or worse, match a coincidentally-advanced nonce).

**Fix.** `const network = await provider.getNetwork(); if (Number(network.chainId) !== proposal.chainId) throw new Error(…);`

## IN-01: `confirmDeployment.ts` exits `0` when the Safe has not executed yet

**Location:** `scripts/safe/confirmDeployment.ts:129-132`

**Problem.** "Not executed yet" exits with code `0`, so a CI step calling this
script reads success even though nothing was updated. For a poll/confirm script
this is ambiguous.

**Fix.** Consider exit `0` only after a successful update, and a non-zero
(e.g. `2`) "pending" code so callers can distinguish pending from done.

## IN-02: `confirmDeployment.ts` carries the old per-facet `version` on replaced facets

**Location:** `scripts/safe/confirmDeployment.ts:205`

`version: (currentDeployed[name] as any)?.version ?? 0` is taken from the
pre-upgrade entry matched by selector overlap. When a facet is *replaced* (new
address, same selectors) the per-facet `version` is not bumped even though the
bytecode changed; only the top-level `protocolVersion` moves. Confirm whether
per-facet `version` is meant to track code-version independently of
`protocolVersion`; if not, bump it alongside `protocolVersion`.
